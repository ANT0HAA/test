# AI Конструкторское бюро — Керамические заводы

Мультиагентная платформа для проектирования керамических (кирпичных) заводов.
Каждый агент — отдельный специалист со своей базой знаний. Главный конструктор
(оркестратор) принимает задачи и распределяет их между специалистами.

## Архитектура

```
Контролёр проекта (вы)
        │
        ▼
Главный конструктор (оркестратор, LangGraph router)
        │  делегирует
        ▼
┌──────────────────────────────────────────────────────┐
│ Технолог · Конструктор · Механик · Электрик · КИПиА   │
│ Нормоконтролёр · Сметчик · Документовед                │
└──────────────────────────────────────────────────────┘
        │  у каждого своя коллекция
        ▼
   База знаний (ChromaDB) — поиск по документам агента
```

### Стек
- **Backend:** Python · FastAPI · LangGraph · **Ollama (локальная модель)** · ChromaDB · PostgreSQL
- **Отчёты:** python-docx (DOCX) · openpyxl (XLSX) · reportlab (PDF)
- **Frontend:** React · TypeScript · Vite · Tailwind
- **Связь:** WebSocket (стриминг токенов), REST (загрузка документов, проекты)
- **Модель:** по умолчанию `qwen3` через Ollama (без оплаты токенов, всё локально).
  Провайдер переключается одной строкой в `.env` (`LLM_PROVIDER=ollama|anthropic`).

## Возможности (Фаза 1 — MVP)

- ✅ Оркестратор маршрутизирует задачи или отвечает сам
- ✅ Прямой диалог с любым специалистом
- ✅ База знаний на агента: загрузка PDF / DOCX / XLSX / TXT
- ✅ Семантический поиск по документам подмешивается в контекст агента
- ✅ Стриминг ответов в реальном времени
- ✅ Пакетное наполнение базы знаний папкой документов (`knowledge/seed.py`)
- ✅ Видимость базы знаний в UI: бейдж с числом фрагментов у агента и список
  файлов-источников (`GET /api/knowledge`)

## Возможности (Фаза 2)
- ✅ **Память проектов (PostgreSQL):** история диалога и «запомненные» решения
  сохраняются между сессиями и переживают перезапуск backend
- ✅ Несколько проектов: создание и переключение, у каждого — своя история
- ✅ «Запомненные» решения проекта подмешиваются в контекст агента наряду
  с базой знаний. Сохранить решение: написать в чат `запомни: <текст>`
  (привязывается к активному агенту) или через `POST /api/projects/{id}/memory`
- ✅ **Межагентное взаимодействие:** на сложный запрос Главный конструктор
  строит план из нескольких специалистов; они отвечают последовательно
  (каждый — отдельным «пузырём»), видя результаты предыдущих. После
  специалистов автоматически включается нормоконтроль; при замечаниях —
  до 2 итераций доработки. Граф защищён от зацикливания жёстким лимитом
  шагов. На слабой локальной модели — откат на одного агента по ключевым
  словам (см. `_keyword_route`, `_sanitize_plan` в `graph/graph.py`)
- ✅ **Генерация отчётных документов:** по проекту формируются
  пояснительная записка (**DOCX**), ведомость оборудования (**XLSX**)
  и сводный отчёт (**PDF**) — кнопка «Сформировать документ» в шапке чата
  или `POST /api/export` (`{project_id, doc_type}`). Документы собираются
  из истории работы агентов, с титулом и основной надписью (штампом).
  Позиции ведомости извлекаются моделью с откатом на эвристику (работает
  и без Ollama). Модуль — `backend/export/`
- ✅ **Интеграция с Компас-3D** (отдельный Windows-сервис `kompas-connector`):
  чтение `.cdw`/`.frw` (штамп, тексты, размеры → JSON) и генерация чертежей через
  COM API — план/габарит (`kind=foundation`) и **генплан завода** (`kind=site_plan`:
  корпуса с подписями и штампом). Также `POST /api/kompas/design {brief}` —
  **Конструктор (LLM) предлагает состав корпусов по брифу, Компас рисует генплан**
  (откат на типовой состав). Backend проксирует (`/api/kompas/status|read|generate|design`)
  и **работает без коннектора**. См. `kompas-connector/README.md`
- ✅ **Мультиотраслевость:** «отрасль» над агентами — у каждой свой набор
  специалистов и изолированная база знаний (коллекции `{industry}_{agent}`).
  Отрасль выбирается при создании проекта. По умолчанию — керамика; включена
  вторая отрасль «Заводы ЖБИ» как пример. См. «Как добавить новую отрасль»
- ✅ **Управление агентами и отраслями из UI** (кнопка-шестерёнка в сайдбаре):
  добавить/удалить отрасль, добавить/изменить/сбросить агента (название,
  описание, цвет, иконка, системный промпт, ключевые слова). Встроенные агенты
  редактируются как оверрайды и сбрасываются к стандарту; кастомные — удаляются.
  Изменения хранятся в PostgreSQL поверх встроенных дефолтов, граф пересобирается
  автоматически. REST: `POST/DELETE /api/industries`, `PUT/DELETE /api/agents/{industry}/{agent}`

## Возможности (Фаза 3 — расчётное ядро, лаборатория, многопользовательский режим)
- ✅ **Расчётное ядро (`backend/calc/`)** — детерминированные инженерные расчёты по
  методикам из документов заказчика (НЕ выдумки модели): производственная программа
  и потребность ресурсов, теплотехнический расчёт сушила, подбор оборудования,
  электрические нагрузки/КТП, площади корпусов, смета себестоимости, оксидный состав
  шихты. Тесты воспроизводят числа документов (себестоимость 2255.5 ₽/1000 и др.).
  REST: `POST /api/calc/{production,dryer,equipment,electrical,areas,estimate,shihta,lab}`
- ✅ **Спецификация проекта — единый источник истины** (`build_spec`): из исходных
  данных собирается структура (программа, ресурсы, оборудование, электрика, корпуса,
  себестоимость). Окно «Спецификация» (значок документа); из неё же берёт цифры экспорт
  документов. `GET /api/projects/{id}/spec`
- ✅ **Генплан по вычисленным площадям**: корпуса для чертежа Компас берутся из
  расчётного ядра (`POST /api/projects/{id}/site-plan`), а не задаются вручную
- ✅ **Модуль «Лаборатория»** (`backend/calc/lab.py`, окно с колбой) — расчёты по сырью
  на стандартных методиках: число пластичности и группа сырья (ГОСТ 21216), усреднение
  шихты (любое число глин), решение «отощитель ↔ удлинение сушилки» с дозировкой песка,
  чувствительность к сушке, выработка карьера (добыча/вскрыша/срок), усреднительный
  штабель, подбор питателей с ограничением, схема по способу формования, контрольные
  точки опробования. Целевая пластичность и нагрузки считаются системой; глины и их
  свойства берутся из отчёта лаборатории (`POST /api/projects/{id}/lab`, LLM-извлечение).
  Каталоги питателей и часть коэффициентов — типовые плейсхолдеры до данных заказчика
- ✅ **Распознавание скан-PDF (OCR)** в материалах проекта: PDF без текстового слоя
  автоматически прогоняется через Tesseract (`backend/knowledge/ocr.py`); распознанный
  текст становится фрагментами проекта
- ✅ **Данные проекта (материалы)** — окно «Данные проекта» (значок папки): список
  распознанных фрагментов с **правкой и удалением** (по фрагменту и по файлу).
  `GET/PATCH/DELETE /api/projects/{id}/materials`
- ✅ **Разбор отчёта лаборатории** (LLM): извлечение компонентов и оксидов → состав
  массы (`POST /api/projects/{id}/analyze-lab`)
- ✅ **Интерактивные уточняющие формы**: на запрос проектирования без обязательных
  данных бюро запрашивает их окошком (событие `clarify` по WebSocket), затем повторяет
  запрос; можно проектировать без уточнения
- ✅ **Авторизация (логин/пароль + роли)**: проекты привязаны к владельцу и видны
  только ему (админ — все). Первый зарегистрированный пользователь — администратор,
  управляет пользователями. `POST /api/auth/{register,login,logout}`, `/api/users`.
  Пароли — PBKDF2 (стандартная библиотека), токен сессии в заголовке/в query для WS
- ✅ **Версионирование артефактов**: при каждой генерации (документ, ZIP-пакет, генплан)
  сохраняется версия — файл + снимок спецификации; история со скачиванием (значок
  часов). `GET /api/projects/{id}/versions[/{id}]`
- ✅ **Полный пакет проекта (ZIP)** и удаление проекта

## Быстрый старт (локально, Windows) — проверено

Полный сценарий, на котором проект проверен end-to-end на живой модели:

1. **PostgreSQL** — запустить службу (нужны права администратора):
   `Start-Service postgresql-x64-16` (или через «Службы»). База/роль `bureau`
   создаются один раз (см. раздел про PostgreSQL ниже).
2. **Ollama** — `ollama serve`, модель: `ollama pull qwen3:14b`
   (для слабого железа сгодится `qwen3:4b`).
3. **backend/.env** (создать, gitignored):
   ```
   LLM_PROVIDER=ollama
   LLM_MODEL=qwen3:14b
   LLM_REASONING=false
   DATABASE_URL=postgresql+asyncpg://bureau:bureau@localhost:5432/bureau
   # OCR скан-PDF (опц.): путь к Tesseract определяется автоматически;
   # OCR_MAX_PAGES=0 — распознавать все страницы (по умолчанию 60)
   OCR_MAX_PAGES=0
   ```
4. **Backend**: `cd backend && .venv\Scripts\activate && uvicorn main:app --port 8000`
5. **Frontend**: `cd frontend && npm run dev` → http://localhost:5173
6. (опц.) **Наполнить базу знаний** документами: см. раздел ниже (`knowledge/seed.py`).

> Важно для qwen3 (reasoning-модель): держите `LLM_REASONING=false` — иначе
> модель тратит весь бюджет токенов на «думанье» и не доходит до ответа.

## Запуск для разработки (локально)

### 0. Ollama (модель)
```bash
# установить Ollama: https://ollama.com/download
ollama serve                 # запустить сервер (обычно стартует сам)

# скачать модель — выберите по своему железу:
ollama pull qwen3:235b       # максимум, нужен сервер с большой VRAM
# ollama pull qwen3:30b      # баланс, ~24 ГБ
# ollama pull qwen3:14b      # ~12 ГБ
# ollama pull qwen3:8b       # ~8 ГБ
```

### 1. PostgreSQL (память проектов)
```bash
# вариант с Docker:
docker compose up -d postgres
# либо локальный Postgres — создайте роль и базу, совпадающие с DATABASE_URL:
#   CREATE ROLE bureau LOGIN PASSWORD 'bureau';
#   CREATE DATABASE bureau OWNER bureau;
```
Таблицы создаются автоматически при старте backend (`init_db`).

### 2. Backend
```bash
cd backend
python -m venv .venv && source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt

cp ../.env.example ../.env     # LLM_PROVIDER=ollama, выставьте LLM_MODEL под скачанный тег,
                               # при необходимости поправьте DATABASE_URL
uvicorn main:app --reload --port 8000
```

### 3. Frontend
```bash
cd frontend
npm install
npm run dev          # откроется http://localhost:5173
```

> В шапке чата отображается активная модель. Если модель не скачана или
> Ollama не запущен — сверху появится жёлтая плашка с подсказкой что сделать.

## Запуск через Docker (для переноса на сервер)
```bash
cp .env.example .env        # выставьте LLM_MODEL под VRAM (напр. qwen3:14b)
docker compose up -d ollama postgres
docker compose exec ollama ollama pull qwen3:14b    # или иную версию под VRAM
docker compose up --build
# frontend → http://localhost:3000
# backend  → http://localhost:8000
```
В compose Ollama поднимается как отдельный сервис. Для работы на GPU
раскомментируйте блок `deploy.resources` в `docker-compose.yml`
(нужен NVIDIA Container Toolkit).

Особенности контейнеризации:
- **OCR скан-PDF работает «из коробки»** — в образ backend ставится Tesseract
  с русским языком (`tesseract-ocr`, `tesseract-ocr-rus`); путь определяется
  автоматически (`backend/knowledge/ocr.py`).
- **Компас-3D НЕ контейнеризуется** (нужен Windows + COM + установленный Компас).
  `kompas-connector` запускается отдельно на Windows-машине, backend ходит к нему
  по сети (`KOMPAS_CONNECTOR_URL`). Без него платформа работает, чертежи — по запросу.
- Данные (Chroma, загрузки, БД) хранятся в `./data` (volumes), переживают пересборку.
- `.dockerignore` исключает `.venv`, кэши, локальные данные и `.env` из образов.

## Безопасность (перед публикацией на сервер)
Авторизация: логин/пароль (**PBKDF2-SHA256**, соль, сравнение `hmac.compare_digest`),
сессионные токены (`secrets`, срок 30 дней, проверка истечения). Проекты изолированы
по владельцу; админ-эндпоинты под `require_admin`; WebSocket требует токен.

Применено по итогам ревью:
- **Публичная регистрация закрыта** после создания первого администратора —
  новых пользователей заводит админ (флаг `ALLOW_OPEN_REGISTRATION=true` открывает
  сам/регистрацию, если нужно).
- **CORS** берётся из `CORS_ORIGINS` (на сервере укажите конкретные домены вместо
  `*`); `allow_credentials` выключен — авторизация через Bearer-заголовок, не cookies.

Рекомендуется доусилить перед выходом «в интернет»:
- **HTTPS/обратный прокси** (nginx/traefik) перед backend; тогда токен в URL
  WebSocket-соединения не утекает в логи в открытом виде.
- **Рейт-лимит** на `/api/auth/login` (защита от перебора) и политику паролей.
- Legacy-проекты без владельца (`owner_id IS NULL`) доступны любому вошедшему —
  на чистой БД таких нет; при миграции старых данных назначьте владельцев.

## Как добавить нового агента
Вариант А (через UI, без перезапуска): кнопка-шестерёнка в сайдбаре →
«добавить агента» / редактировать существующего. Изменения сохраняются в БД,
граф пересобирается автоматически.

Вариант Б (встроенный дефолт в коде): добавьте запись в набор агентов нужной
отрасли в `backend/agents/definitions.py` (`_BUILTIN_INDUSTRIES[...]["agents"]`):
id, display_name, description, color, icon, system_prompt; при желании —
ключевые слова в `routing_keywords` отрасли. Перезапустите backend.

## Как добавить новую отрасль
1. Добавьте запись в `INDUSTRIES` в `backend/agents/definitions.py`:
   `{id, display_name, agents: {...}, routing_keywords: {...}}`.
   В наборе агентов желателен `orchestrator` и `norm_control`
   (тогда работает межагентный нормоконтроль).
2. Перезапустите backend. Отрасль появится в `/api/industries`,
   её агенты — в `/api/agents?industry=<id>`, база знаний изолируется
   в коллекциях ChromaDB вида `{industry}_{agent}`.
3. На фронтенде отрасль выбирается при создании проекта; набор агентов
   и база знаний привязаны к отрасли активного проекта. «Керамика» —
   отрасль по умолчанию (`DEFAULT_INDUSTRY`).

Пример второй отрасли уже включён: «Заводы ЖБИ и товарного бетона»
(`concrete`) со своим набором специалистов.

## Наполнение базы знаний документами
Через UI: кнопка «Обучить документом» — загрузка PDF / DOCX / XLSX / TXT в базу
выбранного агента (с учётом активной отрасли).

Пакетно (целая папка) — скрипт `backend/knowledge/seed.py`: раскладывает файлы
по агентам отрасли согласно маппингу «имя файла → агент(ы)» в самом скрипте:
```bash
cd backend && source .venv/Scripts/activate   # Windows: .venv\Scripts\activate
python -m knowledge.seed "C:\путь\к\Документы" --industry ceramics --max-pdf-pages 0
```
Поддерживаются `txt / pdf / docx / xlsx`, архивы `rar / 7z / zip` (через 7-Zip,
рекурсивно), а также старый `.doc` — если установлен Microsoft Word (конвертация
через COM на Windows). `--max-pdf-pages 0` — весь документ; иначе первые N страниц.

**OCR скан-PDF** (карты залеганий, геологические отчёты — без текстового слоя).
Питон-зависимости (`pymupdf`, `pytesseract`, `pillow`) уже в `requirements.txt`;
дополнительно нужен системный **Tesseract** с языковым пакетом `rus` (путь к
`tesseract.exe` определяется автоматически; русский язык — `backend/.tessdata`).
- В **материалах проекта** (кнопка «+») скан-PDF распознаётся **автоматически**:
  если текстового слоя нет, запускается OCR (`backend/knowledge/ocr.py`).
- В **пакетном наполнении базы знаний** OCR включается флагом `--ocr`:
  ```bash
  python -m knowledge.seed "C:\путь\к\Документы" --ocr --max-pdf-pages 0
  ```
Без Tesseract OCR мягко отключается (выдаётся понятная ошибка/сканы пропускаются).
Лимит страниц на OCR — `OCR_MAX_PAGES` в `.env` (0 — все).

- **Ожидает данных заказчика:** обратный подбор пропорций глин под целевой состав,
  реальные каталоги питателей и точные коэффициенты лаборатории.

## Замечания о модели и «обучении»
Платформа работает на локальной модели через Ollama — токены не оплачиваются,
данные проектов не покидают вашу машину/сервер. Маршрутизатор (Главный
конструктор) использует structured output; на локальных моделях он иногда
отдаёт невалидный JSON, поэтому предусмотрен откат на маршрутизацию по
ключевым словам (`_keyword_route` в `graph/graph.py`) — система не падает.

«Обучение» = пополнение базы знаний документами + системные промпты.
Это retrieval-подход: агент не меняет веса модели, а получает релевантные
фрагменты из своей базы в контекст. Дообучение (fine-tuning) самой модели —
это отдельный этап, который имеет смысл, когда накопятся свои примеры
по проектам; архитектура к переключению модели готова.

Чтобы вернуться на облако Claude для сравнения качества — в `.env`:
`LLM_PROVIDER=anthropic`, `LLM_MODEL=claude-sonnet-4-6`, задать `ANTHROPIC_API_KEY`.
