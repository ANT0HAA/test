# Сервер для AI Конструкторского бюро (и не только)

Гайд по серверу, на котором проект работает на **максимальной модели** (`qwen3:235b`)
и остаётся запас под другие задачи. Главное требование — не диск и не CPU, а
**видеопамять (VRAM) под LLM**. Детерминированное ядро (расчёты, документы, генплан)
почти ничего не требует — тяжёлая только модель.

## Сколько ресурсов нужно модели

| Модель Ollama | Активных параметров | ~Размер Q4 | Нужно VRAM (с контекстом) | Скорость на адекватном GPU |
|---|---|---|---|---|
| `qwen3:235b` (MoE A22B) | 22 млрд из 235 | ~140 ГБ | **~150–170 ГБ** | 20–50 ток/с |
| `qwen3:30b` / `32b` | 30–32 млрд | ~18–20 ГБ | ~24–32 ГБ | 40–80 ток/с |
| `qwen3:14b` | 14 млрд | ~9.6 ГБ | ~12–16 ГБ | 40–60 ток/с |
| `qwen3:8b` | 8 млрд | ~5 ГБ | ~8 ГБ | 60–100 ток/с |

Вывод: «максимальная» `235b` требует **~150+ ГБ VRAM**, то есть несколько GPU
или профессиональные карты. Но `qwen3:30b/32b` по качеству близки к 235b на 90%
задач и помещаются на **одну** карту 48 ГБ — это «золотая середина».

## Вариант 1. Облако (рекомендую начать с него)

Без капитальных вложений — аренда GPU по часам. Идеально «попробовать 235b» и понять,
нужна ли своя железка.

- **Провайдеры:** RunPod, Vast.ai, Lambda Cloud, immers.cloud / selectel (РФ).
- **Под `qwen3:235b`:** 2×H100 80GB или 4×A100 80GB — ~4–10 $/час (≈ 350–900 ₽/час).
- **Под `30b/70b`:** 1×A100 80GB или 1×H100 — ~1.5–3 $/час.
- Поднимаешь Ubuntu-инстанс с GPU, ставишь Docker + этот проект (раздел «Софт»),
  тянешь модель — и работаешь. Выключил инстанс — не платишь.

**Плюс:** старт за час, ноль капекса, легко сменить размер.
**Минус:** платишь, пока работает; данные в облаке.

## Вариант 2. Рабочая станция (баланс, ~5–9 тыс. $ / 500–850 тыс. ₽)

Своя машина под `qwen3:30b/32b` на полной скорости и «много всего» (разработка,
другие модели, базы, контейнеры). `235b` пойдёт с выгрузкой части слоёв в RAM
(медленно, но работает — MoE активирует лишь 22B).

| Узел | Рекомендация |
|---|---|
| **GPU** | 1× **NVIDIA RTX 6000 Ada 48 ГБ** (идеал) или 1× **RTX 5090 32 ГБ** (дешевле) или 2× RTX 3090/4090 24 ГБ (NVLink не нужен, ollama делит слои) |
| **CPU** | AMD Ryzen 9 7950X / Threadripper 7960X (24+ ядра — под «много всего») |
| **RAM** | **128 ГБ** DDR5 (для выгрузки 235b желательно 256 ГБ) |
| **Диск** | NVMe **2 ТБ** Gen4 под модели/данные (модели быстро читаются — важно для загрузки в VRAM) |
| **БП/корпус** | 1000–1200 Вт (80+ Gold), хороший обдув |
| **ОС** | Ubuntu Server 24.04 LTS |

На 48 ГБ карте `qwen3:32b` идёт быстро; `qwen3:14b` — мгновенно. Это покрывает чат
агентов «вживую» + остаётся ресурс под параллельные задачи.

## Вариант 3. Сервер максимум — `qwen3:235b` на скорости ($25–60 тыс.)

Когда нужна именно 235b интерактивно и большой запас.

| Узел | Рекомендация |
|---|---|
| **GPU** | **4× RTX 6000 Ada 48 ГБ** (192 ГБ суммарно) — оптимум цена/VRAM; или **2× H100 80GB**; или 2× RTX 6000 Pro Blackwell 96 ГБ |
| **CPU** | AMD EPYC 9004 (32–64 ядра) или Threadripper PRO |
| **RAM** | **256–512 ГБ** DDR5 ECC |
| **Диск** | 2× NVMe Gen4 4 ТБ (RAID1 под модели/данные) + HDD под бэкапы |
| **Платформа** | Серверная MB с 4× PCIe x16, БП 2× 1600 Вт, стоечный/башенный корпус с продувом |
| **ОС** | Ubuntu Server 24.04 LTS |

192 ГБ VRAM хватает на `qwen3:235b` Q4 с контекстом + место под вторую модель/эмбеддинги.
Запас CPU/RAM — под «ещё много всего»: дообучение мелких моделей, пакетные расчёты,
несколько сервисов, БД, CI.

> Дешёвый компромисс под 235b «медленно, но своё»: 1× 48 ГБ GPU + **256 ГБ RAM** —
> MoE-модель выгружается в RAM, скорость ~5–10 ток/с (для не-интерактивных задач ок).

## Софт-настройка сервера (Ubuntu) — одинаково для облака и своего железа

```bash
# 1. Драйвер NVIDIA (под свою карту) и проверка
sudo ubuntu-drivers autoinstall && sudo reboot
nvidia-smi                       # должна показать GPU

# 2. Docker + Compose
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER    # перелогиниться

# 3. NVIDIA Container Toolkit (GPU внутрь контейнеров)
curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg
curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list | \
  sed 's#deb #deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] #' | \
  sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list
sudo apt update && sudo apt install -y nvidia-container-toolkit
sudo nvidia-ctk runtime configure --runtime=docker && sudo systemctl restart docker
docker run --rm --gpus all nvidia/cuda:12.4.0-base-ubuntu22.04 nvidia-smi   # проверка GPU в контейнере

# 4. Проект
git clone https://github.com/ANT0HAA/test.git ceramic-bureau && cd ceramic-bureau
cp .env.example .env
#   в .env: LLM_MODEL=qwen3:235b (или 30b/14b под VRAM); CORS_ORIGINS=<домен>
#   GPU-блок для ollama в docker-compose.yml уже включён (deploy.resources nvidia)

docker compose up -d ollama postgres
docker compose exec ollama ollama pull qwen3:235b      # или нужный размер
docker compose up -d --build                            # фронт :3000, бэкенд :8000
```

На Linux-сервере GPU-проброс работает «из коробки» (в отличие от Windows/WSL2, где
компьют через прослойку может тормозить — что мы и видели на текущей машине).

## Перед публикацией «в интернет»
- Обратный прокси с **HTTPS** (nginx/Caddy/Traefik) перед `:3000`.
- В `.env`: `CORS_ORIGINS=https://твой-домен`, `ALLOW_OPEN_REGISTRATION=false`.
- Фаервол: наружу только 80/443; 8000/5433/11434 — только локально.
- Бэкап тома `./data` (Postgres/Chroma/uploads) по расписанию.
- Компас-3D остаётся на **отдельной Windows-машине** (COM не контейнеризуется),
  сервер ходит к коннектору по сети через `KOMPAS_CONNECTOR_URL`.

## Что выбрать
- **Просто попробовать 235b / непостоянная нагрузка** → Вариант 1 (облако).
- **Ежедневная работа, быстрый чат 30b, плюс прочие задачи** → Вариант 2 (станция, 48 ГБ GPU).
- **235b интерактивно + большой запас** → Вариант 3 (мульти-GPU сервер).

Цены ориентировочные (на начало 2026), уточняются у поставщиков. По любому из
вариантов помогу выбрать точные комплектующие и поднять проект.
