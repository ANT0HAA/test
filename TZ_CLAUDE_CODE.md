## 1. Что это за проект

Мультиагентная платформа для проектирования керамических (кирпичных) заводов.
Главный конструктор (оркестратор) принимает задачу от пользователя, решает —
ответить самому или делегировать одному из 8 агентов-специалистов. У каждого
агента своя база знаний (документы), в которую можно загружать ГОСТы,
техкарты, инструкции по проектам.

**Принципиальные требования заказчика (не нарушать):**

1. Платформа полностью независимая и локальная. Модель — **Ollama**, без оплаты
   токенов и без отправки данных наружу. Облако (Anthropic) допустимо только как
   опциональный провайдер для сравнения качества.
2. Расширяемость: добавление нового агента и новой отрасли не должно требовать
   переписывания ядра.
3. «Обучение» агента = пополнение его базы знаний документами + системный промпт
   (retrieval-подход). Дообучение весов модели — вне текущего объёма.
4. Пользователь выступает контролёром проекта: он видит, кто из агентов
   отвечает, и может общаться с каждым напрямую.

**Целевое железо:** сейчас разработка на RTX 3080 Ti (12 ГБ VRAM), модель
`qwen3:14b`. Позже — отдельный сервер, модель `qwen3:30b` / `qwen3:235b`.
Код не должен быть завязан на конкретную модель: только на `LLM_PROVIDER` и
`LLM_MODEL` из конфига.

## 2. Технологический стек

| Слой        | Технологии |
|-------------|------------|
| Backend     | Python 3.12, FastAPI, LangGraph, ChromaDB |
| LLM         | Ollama (`langchain-ollama`), опц. Anthropic (`langchain-anthropic`) |
| Frontend    | React 18, TypeScript, Vite, Tailwind, lucide-react |
| Транспорт   | WebSocket (стриминг токенов), REST (загрузка документов, статус) |
| Хранилище   | ChromaDB (база знаний), файловая система (загрузки) |
| Деплой      | Docker Compose (ollama + backend + frontend) |

Планируемое к добавлению в фазе 2: PostgreSQL (память проектов),
интеграция с Компас-3D (COM API на Windows), генерация DOCX/PDF/XLSX.

## 3. Текущее состояние (готово, фаза 1 / MVP)

- 9 агентов с детальными промптами по керамике — `backend/agents/definitions.py` (`AGENTS`).
- Граф LangGraph — `backend/graph/graph.py`:
  - `_llm(streaming)` — фабрика модели по `LLM_PROVIDER` (ollama|anthropic).
  - `orchestrator_router` — маршрутизация через structured output
    (`RoutingDecision`) с **откатом на `_keyword_route`** при невалидном JSON.
  - `orchestrator_respond` — прямой ответ оркестратора.
  - `_make_specialist(agent_id)` — фабрика нод специалистов.
  - `build_graph` / `get_graph` — сборка и синглтон скомпилированного графа.
  - `BureauState` — состояние графа.
- База знаний — `backend/knowledge/chroma.py`:
  - коллекция на агента (`get_collection`), чанкинг, дедуп по id,
    `add_text` / `add_file` (txt/pdf/docx), `search` (семантический поиск).
- API — `backend/main.py`:
  - `WS /ws/{session_id}` — чат со стримингом (`astream_events`, v2).
  - `POST /api/upload` — загрузка документа в базу агента.
  - `GET /api/agents`, `GET /api/health`, `GET /api/llm-status`,
    `DELETE /api/session/{id}`.
  - История сессий — **in-memory** (`_sessions`), обрезка по `MAX_HISTORY`.
- Frontend — `frontend/src/`: `Sidebar`, `ChatPanel`, `UploadModal`, `App`,
  `api/client.ts` (REST + WebSocket-хук). Плашка статуса модели.

**Известные ограничения, которые фаза 2 должна снять:**
- Межагентного взаимодействия нет: путь всегда «оркестратор → один специалист → ответ».
- Память только на время сессии (теряется при перезапуске).
- Нет работы с чертежами и генерации отчётных документов.

## 4. Архитектурные правила (обязательны к соблюдению)

1. **Никаких прямых обращений к Anthropic/Ollama в коде нод.** Любой вызов
   модели — только через `_llm()` из `graph/graph.py`. Это единственная точка,
   знающая о провайдере.
2. **Локальная модель — первичный сценарий.** Любая новая логика, зависящая от
   модели, должна корректно работать на `qwen3:14b` и иметь деградацию/откат,
   если модель вернула невалидный результат (по образцу `_keyword_route`).
3. **Добавление агента = одна запись в `AGENTS`.** Граф, маршрутизация, UI и
   база знаний обязаны подхватывать нового агента без правок в других местах.
   Если задача требует ручной правки в нескольких местах при добавлении агента —
   это повод отрефакторить, а не размазывать.
4. **База знаний изолирована по агентам.** Имя коллекции — `agent_{id}`. Не
   смешивать знания разных агентов без явного требования.
5. **Стриминг идёт только из конечных нод** (`orchestrator_respond`, ноды
   специалистов). Маршрутизатор не стримит.
6. **Секреты — только через `.env`** (`config.py` / pydantic-settings). Не
   хардкодить ключи, URL, пути.
7. **Обратная совместимость API.** Существующие эндпоинты и формат
   WebSocket-сообщений (`agent_start` / `token` / `done` / `error`) не ломать
   без обновления фронтенда в той же задаче.

## 6. Соглашения по коду

**Python**
- Типизация обязательна (аннотации типов на всех публичных функциях).
- Асинхронность: ноды графа и эндпоинты — `async`. Крупные блокирующие операции
  выносить в `run_in_executor`, если начнут тормозить event loop.
- Имена: функции/переменные `snake_case`, классы `PascalCase`,
  «приватные» хелперы — с префиксом `_`.
- Комментарии и docstring — на русском, лаконично, по делу.
- Никаких `print` для логики — использовать `logging`.

**TypeScript / React**
- Функциональные компоненты + хуки. Без классов.
- Состояние — `useState` / `useReducer`. **Не использовать** `localStorage` /
  `sessionStorage` без явного требования.
- Стили — Tailwind, токены из `tailwind.config.js` (`ink`, `clay`, `muted`, `faint`).
- Тексты интерфейса — на русском.

**Общее**
- Каждая задача — отдельная ветка и осмысленные коммиты.
- Перед завершением задачи прогнать проверки из раздела 8.

## 7. Бэклог фазы 2 (приоритизированный)

Брать задачи сверху вниз. Каждая — самостоятельная, с критериями приёмки.
Если задача крупная — декомпозировать на подзадачи в рамках одной ветки.

### Задача 1 — Память проектов (PostgreSQL) [приоритет: высокий]

**Зачем:** агенты должны помнить решения по проектам между сессиями
(«запоминать инструкции и использовать их позже» — требование заказчика).

**Объём:**
- Поднять PostgreSQL как сервис в `docker-compose.yml` (+ том для данных).
- Слой доступа к БД (SQLAlchemy async или asyncpg) — новый модуль
  `backend/storage/db.py`.
- Сущности: `Project` (id, name, industry, created_at),
  `ProjectMessage` (project_id, role, agent, content, created_at),
  `ProjectMemory` (project_id, agent, key, value) — для «запомненных» решений.
- Заменить in-memory `_sessions` на персистентную историю по `project_id`
  (с сохранением обрезки контекста для модели).
- REST: создать / список / получить проект; переключение активного проекта.
- Frontend: селектор проекта в шапке, создание нового проекта.

**Критерии приёмки:**
- После перезапуска backend история и память проекта сохраняются.
- Агент в ответе может опираться на ранее «запомненные» решения проекта
  (память подмешивается в контекст наряду с базой знаний).
- Добавление проекта не ломает существующий чат-флоу.

---

### Задача 2 — Межагентное взаимодействие [приоритет: высокий]

**Зачем:** сейчас оркестратор делегирует ровно одному специалисту. Нужно,
чтобы агенты передавали задачи друг другу и итерировали, а нормоконтролёр
мог проверять результат до выдачи пользователю.

**Объём:**
- Расширить `BureauState`: план работ (список подзадач с исполнителями),
  накопленные результаты по агентам, счётчик итераций.
- Оркестратор формирует план (несколько агентов) вместо одного выбора.
  Сохранить откат на `_keyword_route`, если модель не вернёт валидный план.
- Граф: последовательное / условное прохождение агентов по плану; после
  специалистов — узел `norm_control` как проверяющий; при замечаниях —
  возврат на доработку (цикл с лимитом итераций, напр. ≤2).
- Стриминг: в UI должно быть видно смену агентов (использовать существующий
  `agent_start`); каждый агент — отдельный «пузырь» в чате.
- Защита от зацикливания: жёсткий лимит шагов графа.

**Критерии приёмки:**
- На запрос вида «спроектируй обжигательный корпус» в работу включаются
  несколько агентов (напр. конструктор + механик + нормоконтролёр), и
  пользователь видит их последовательные ответы.
- При замечаниях нормоконтролёра происходит ≤2 итерации доработки, затем
  выдаётся финальный результат.
- Граф не зацикливается ни при каком вводе (есть лимит).

---

### Задача 3 — Генерация отчётных документов (DOCX / PDF / XLSX) [приоритет: средний]

**Зачем:** заказчику нужны отчётные документы и ведомости на выходе.

**Объём:**
- Модуль `backend/export/` с генераторами: пояснительная записка → DOCX,
  спецификация / ведомость оборудования → XLSX, сводный отчёт → PDF.
- Использовать `python-docx`, `openpyxl`, библиотеку для PDF.
- Оформление по ЕСКД / СПДС минимально: титул, основная надпись (штамп),
  нумерация — согласуется с агентом `documentalist`.
- REST `POST /api/export` — на основе истории / результата проекта собрать
  документ выбранного типа, вернуть файл на скачивание.
- Frontend: кнопка «Сформировать документ» с выбором типа.

**Критерии приёмки:**
- Из результата работы агентов формируется корректный DOCX / XLSX / PDF,
  открывается в Word / Excel / просмотрщике без ошибок.
- Ведомость оборудования выгружается в XLSX с колонками (наименование,
  марка, кол-во, ед., примечание).

### Задача 4 — Интеграция с Компас-3D [приоритет: средний, зависит от Windows]

**Зачем:** чтение существующих чертежей и генерация новых по заданию.

**Важно:** Компас-3D управляется через COM API и работает **только на Windows**.
Эта задача выполняется в Windows-окружении, не в Docker-контейнере backend.
Спроектировать как отдельный сервис-коннектор.

**Объём:**
- Модуль / сервис `kompas-connector` (Python + `pywin32`), общается с backend по HTTP.
- Чтение `.cdw` / `.frw`: извлечь спецификацию, размеры, текстовые поля.
- Генерация простого чертежа по параметрам (напр. план фундамента / раскладка)
  через Kompas COM API.
- REST на стороне backend для проксирования запросов к коннектору
  (с обработкой случая «коннектор недоступен»).

**Критерии приёмки:**
- Загруженный `.cdw` парсится: backend получает структурированные данные.
- По заданию агента генерируется новый `.cdw` с базовой геометрией и штампом.
- Если коннектор не запущен (например, backend на Linux) — внятная ошибка,
  остальная платформа работает.

### Задача 5 — Задел под мультиотраслевость [приоритет: низкий, фаза 3]

**Зачем:** платформа должна расширяться на другие отрасли, не только керамику.

**Объём:**
- Ввести понятие «отрасль» над агентами: namespace коллекций ChromaDB
  (`{industry}_{agent}`) и набор `AGENTS` на отрасль.
- Рефактор `definitions.py`: реестр отраслей → наборы агентов.
- API / Frontend: выбор активной отрасли; «керамика» — отрасль по умолчанию.

**Критерии приёмки:**
- Можно зарегистрировать вторую отрасль с собственным набором агентов и
  изолированной базой знаний, не затрагивая керамику.

## 8. Как запускать и проверять

**Запуск (dev):**

    # модель
    ollama serve
    ollama pull qwen3:14b

    # backend
    cd backend && python -m venv .venv && source .venv/bin/activate
    pip install -r requirements.txt
    cp ../.env.example ../.env   # LLM_PROVIDER=ollama, LLM_MODEL=qwen3:14b
    uvicorn main:app --reload --port 8000

    # frontend
    cd frontend && npm install && npm run dev

**Обязательные проверки перед завершением любой задачи:**
- Backend импортируется и поднимается без ошибок: `uvicorn main:app` стартует.
- `GET /api/health` и `GET /api/llm-status` отвечают 200.
- Сквозной сценарий: отправить сообщение оркестратору и прямому агенту —
  оба отвечают стримингом.
- Загрузка документа в базу знаний агента работает, и агент использует
  загруженный контекст в ответе.
- Frontend собирается: `npm run build` без ошибок TypeScript.
- Если менялся граф — проверить, что нет зацикливания на разнородных вводах.
- Если добавлялись зависимости — обновлён `requirements.txt` / `package.json`.

## 10. Чего НЕ делать

- Не привязывать код к конкретной модели или к Anthropic API в обход `_llm()`.
- Не выносить данные проектов во внешние сервисы / облака.
- Не ломать формат WebSocket-сообщений без синхронной правки фронтенда.
- Не вводить `localStorage` / `sessionStorage` в компонентах без нужды.
- Не дублировать описание агентов вне `AGENTS`.
- Не добавлять тяжёлые зависимости ради мелкой задачи без обоснования.
- Не реализовывать дообучение (fine-tuning) модели — это вне текущего объёма.
